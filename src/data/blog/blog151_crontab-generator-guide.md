写 crontab 最容易犯的错误，是把字段顺序搞错。

`0 9 * * 1-5` 和 `9 0 * * 1-5`，一个是"每天早上 9 点工作日执行"，一个是"每天凌晨 0 点 09 分工作日执行"——两个表达式看起来很像，但语义完全不同。

crontab 的五个字段（分钟、小时、日、月、星期）顺序和含义需要记忆，即使写了多年，在不常用的场景下仍然容易出错。[在线 Crontab 生成器](https://anyfreetools.com/tools/crontab-generator) 提供可视化界面：通过选择频率、时间点、重复规则，自动生成正确的 crontab 表达式，并实时预览接下来 5 次的执行时间，彻底消除手写出错的风险。

## crontab 五字段语法

crontab 表达式由五个字段组成，用空格分隔：

```
* * * * *
│ │ │ │ └── 星期（0-7，0 和 7 都代表周日）
│ │ │ └──── 月份（1-12）
│ │ └────── 日（1-31）
│ └──────── 小时（0-23）
└────────── 分钟（0-59）
```

记忆口诀：**分时日月周**，从最小单位（分钟）到最大单位（月），最后是星期。

每个字段可以是：
- `*`：任意值（"每一个"）
- 具体数字：`5`（第 5 分钟、5 号、5 月……）
- 范围：`1-5`（1 到 5，闭区间）
- 列表：`1,3,5`（第 1、3、5 个）
- 步长：`*/15`（每隔 15 个单位）
- 组合：`1-5/2`（1 到 5，每隔 2 个，即 1、3、5）

## 特殊符号详解

### `*`（星号）— 任意值

```
* * * * *    每分钟执行一次
0 * * * *    每小时整点执行一次
0 0 * * *    每天凌晨 0 点执行
```

### `/`（斜杠）— 步长

```
*/5 * * * *        每 5 分钟执行一次（0, 5, 10, 15...）
*/15 * * * *       每 15 分钟执行一次
0 */2 * * *        每 2 小时的整点执行（0:00, 2:00, 4:00...）
0 9-17/2 * * *     9 点到 17 点，每 2 小时执行（9:00, 11:00, 13:00, 15:00, 17:00）
```

### `,`（逗号）— 枚举

```
0 9,18 * * *       每天 9:00 和 18:00 各执行一次
0 0 1,15 * *       每月 1 号和 15 号的凌晨 0 点执行
0 0 * * 1,3,5      每周一、三、五的凌晨 0 点执行
```

### `-`（连字符）— 范围

```
0 9-17 * * *       每天 9:00 到 17:00，每小时整点执行
0 0 * * 1-5        每周一到周五的凌晨 0 点执行（工作日）
0 0 * * 6-7        每周六、周日的凌晨 0 点执行（周末）
```

### 组合使用

```
5,35 9-18 * * 1-5     工作日的 9:05、9:35、10:05、10:35... 18:05、18:35 执行
0 */4 * * *           每 4 小时整点执行（0:00, 4:00, 8:00, 12:00, 16:00, 20:00）
30 8 1 * *            每月 1 号 8:30 执行
```

## 常用场景模板

### 场景一：数据库每日备份

需求：每天凌晨 3 点执行备份脚本，流量低谷，避开工作时间。

```
0 3 * * *
```

说明：分钟=0，小时=3，其他全为 `*`（每天、每月、每星期都执行）。接下来执行时间：今天凌晨 3:00、明天凌晨 3:00……

在服务器上添加：

```bash
# 编辑当前用户的 crontab
crontab -e

# 添加任务
0 3 * * * /opt/scripts/backup.sh >> /var/log/backup.log 2>&1
```

`>> /var/log/backup.log 2>&1` 把标准输出和错误都追加到日志文件，方便排查失败原因。

### 场景二：工作日发送日报推送

需求：每周一到周五的早上 9 点发送报告通知。

```
0 9 * * 1-5
```

说明：分钟=0，小时=9，星期=1-5（周一到周五）。周六、周日不触发。

### 场景三：每 15 分钟轮询任务

需求：每隔 15 分钟检查一次任务队列，处理待执行的定时任务。

```
*/15 * * * *
```

接下来执行时间：XX:00、XX:15、XX:30、XX:45，每小时触发 4 次。

### 场景四：每月 1 号发送账单

需求：每月 1 号的上午 10 点发送上月账单。

```
0 10 1 * *
```

说明：分钟=0，小时=10，日=1，月和星期都是 `*`（任意月份，不限星期几）。

### 场景五：每周日凌晨全量同步

需求：每周日凌晨 2 点做一次全量数据同步，工作日只做增量。

```
0 2 * * 0
```

说明：星期=0（周日）。注意：星期字段 0 和 7 都代表周日，`* * * * 0` 和 `* * * * 7` 等价。

### 场景六：一天内多次执行

需求：上午 8 点、中午 12 点、下午 5 点各执行一次心跳检查。

```
0 8,12,17 * * *
```

### 场景七：每季度执行一次

需求：每季度第一天凌晨 0 点执行季度汇总。

```
0 0 1 1,4,7,10 *
```

说明：月份枚举 1（1月）、4（4月）、7（7月）、10（10月）——刚好是四个季度的起始月。

## 在 GitHub Actions 中使用

GitHub Actions 的 `schedule` 触发器使用标准 crontab 语法，但有一个常见陷阱：**时区固定为 UTC**，需要换算本地时间。

```yaml
name: Daily Report

on:
  schedule:
    # 北京时间 9:00 = UTC 1:00
    - cron: "0 1 * * *"
  workflow_dispatch:  # 同时支持手动触发

jobs:
  report:
    runs-on: ubuntu-latest
    steps:
      - name: Run report script
        run: node scripts/daily-report.js
```

工具生成器里可以切换时区，选"UTC"后直接得到 Actions 可用的表达式，不需要自己换算。

GitHub Actions 的最小调度间隔是 5 分钟（`*/5 * * * *`），更高频率的表达式会被限流到每 5 分钟一次。

## Node.js 中使用 cron 表达式

如果你在 Node.js 服务里需要定时任务，`node-cron` 库直接复用 crontab 表达式：

```bash
npm install node-cron
```

```typescript
import cron from "node-cron";

// 每天凌晨 3 点执行备份
cron.schedule("0 3 * * *", () => {
  console.log("Running backup...");
  runBackup();
});

// 每 15 分钟处理队列
cron.schedule("*/15 * * * *", () => {
  processQueue();
});

// 工作日早上 9 点发报告
cron.schedule("0 9 * * 1-5", async () => {
  await sendDailyReport();
});
```

`node-cron` 支持标准 crontab 的五字段格式，也支持扩展的六字段格式（第一位是秒，范围 0-59），在需要秒级精度时使用：

```typescript
// 六字段格式：秒 分 时 日 月 周
// 每 30 秒执行一次
cron.schedule("*/30 * * * * *", () => {
  checkHealth();
});
```

## 在线工具 vs 手写的适用场景

**适合用在线工具生成**：
- 不熟悉 crontab 语法的场景（运维不频繁）
- 需要确认"接下来几次执行时间"的调试场景
- 生成复杂的枚举/范围组合（如 `5,15,25,35,45,55 9-17 * * 1-5`）
- 跨时区换算（本地时间 → UTC）

**适合直接手写**：
- 简单的整点/每日/每周规则（`0 0 * * *` 这类熟悉的模式）
- CI/CD 脚本里的注释记录（已有现成模板）

工具最大的价值不是帮你写表达式，而是帮你**验证**——"接下来 5 次执行时间"这个预览功能能在部署前就发现频率设置错误，比部署后查日志发现任务没按预期跑要省事得多。

## 常见错误对照

```
# 错误：想要每 5 分钟，写成了每分钟的第 5 秒（标准 crontab 无秒字段）
5 * * * *      ← 这是"每小时的第 5 分钟"，不是每 5 分钟

# 正确：每 5 分钟
*/5 * * * *

# 错误：想要工作日，写成了 1-5 但忘记字段顺序
* * * * 1-5    ← 这是正确的（星期字段在最后）
* * 1-5 * *    ← 这是"每月 1 到 5 号"，不是工作日

# 错误：时区未换算，想要北京时间 9 点，写成了 UTC+8 的 9 点
0 9 * * *      ← 在 UTC 时区是 UTC 9 点，北京时间 17 点
0 1 * * *      ← 这才是北京时间早上 9 点（UTC 1 点）
```
